# Check-list Code Review: Workout Log API

---

### 1. Общие принципы

- [ ] **Одна цель на PR:** Pull Request решает одну конкретную задачу.
- [ ] **Понятное описание:** заполнен шаблон PR: понятно **что** сделано, **зачем** и **как** это проверить. PR связан с Issue.
- [ ] **CI/CD "зелёный":** все автоматические проверки (linting, tests) успешно пройдены.

---

### 2. Качество кода и API-дизайна

- [ ] **Корректное именование:** переменные, функции и эндпоинты названы понятно и в едином стиле (например, `get_all_workouts`, а не `getworkouts`).
- [ ] **Следование REST:** эндпоинты используют правильные HTTP-методы (`GET` для получения, `POST` для создания) и статусы ответа (`200 OK`, `201 Created`, `404 Not Found`).
- [ ] **Модели Pydantic:**
    - [ ] Все поля запросов и ответов описаны в Pydantic-моделях.
    - [ ] Используется валидация данных (например, `gt=0` для повторений, `max_length` для названий).
- [ ] **Обработка ошибок:** несуществующие объекты (например, `GET /workouts/{неверный_id}`) корректно возвращают ошибку `404 Not Found`, а не падают с ошибкой `500`.

---

### 3. Тестирование
- [ ] **Позитивные сценарии:** написаны тесты, которые проверяют успешное выполнение всех новых эндпоинтов.
- [ ] **Негативные сценарии:** есть тесты на обработку ошибок (например, запрос с некорректными данными, запрос несуществующего ID).
- [ ] **Полнота тестов:** тесты покрывают основную логику и граничные случаи.

---

### 4. Документация
- [ ] **Авто-документация Swagger:** у всех эндпоинтов в Swagger (`/docs`) есть понятные `summary` и `description`, а модели данных имеют описания полей.
